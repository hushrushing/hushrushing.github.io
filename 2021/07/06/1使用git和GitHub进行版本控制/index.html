使用git和GitHub进行版本控制

1 git的核心概念

仓库（repo）：文件历史的数据库，保存在项目目录下的.git隐藏目录中。

提交：添加到仓库的某一时间点的工作快照。

远程：指向非本机上仓库副本的链接。

合并：将不同的“提交”组合到一起而无需手工组合。

2 GitHub

GitHub是一个云托管项目副本的网站，是一个可以用来保存代码仓库的地方。可以使用git来进行版本控制。

3 使用git进行版本控制

3.1生成仓库

仓库需要在已有目录上生成。可以在本地新建一个文件夹，在该文件夹下通过git的init动作可以将其转化为仓库。

不要将一个repo放在另一个repo中，这样很难管理。应该创建很多个不同的repo

3.2 检查仓库状态

使用status命令来检查仓库的状态：

git status总是会显示当前仓库的状态，并给出下一步骤。

3.3 跟踪项目变更

改动数据仓库，如添加或删除文件等。再次执行git status会显示更改过和未提交的文件。要提交这些更改，需要

（1）使用git add将更改添加到临时存储区

添加到临时存储区则改变了仓库，再次执行git add会显示哪些文件在临时区域。注意：文件名不能包含空格

（2）使用git commit提交更改

其中提交消息是关于该更改的有用信息，如“修补了严重错误”，建议不要超过50个字符。

（3）再次使用git status核对执行结果

再次使用git status应该显示“全部提交完毕”。

3.4 审核本地git

使用git的开发循环是：（1）编辑文件；（2）添加文件；（3）提交更改。一旦有了好的“断点”应该及时提交更改，以随时回到这个断点。如果添加了多余的文件，可以通过

撤销提交，但更改的文件仍在目录中。但是只能撤销最近的提交。

4 在GitHub中存储项目

存储在GitHub上的仓库是本地仓库的副本，也叫origin。将本地仓库链接到GitHub有两个常见的过程：

（1）上传：点击“New Repository”在GitHub上新建一个repo，链接本地的repo到GitHub上新建的repo。

（2）下载：克隆GitHub上的repo副本，再修改和编辑代码。

注意：应该先在本地进行开发工作，再将改动合并到远程副本中

4.1 分支和克隆

由于无法直接修改别人的repo，所以应该先将repo复制到自己的账户中，这个过程叫做分支（Fork）。点击“Fork”按钮进行分支，这样就可以在自己的账户下更改副本，将repo克隆（Clone）到本地进行修改。

更改完成后将repo上传回GitHub中，这个过程叫推送（Push）。

可以通过git remote查看远程位置。

可以通过git pull下载其他人的更改并且融合到本地仓库中，这个过程叫拉取（Pull）

注意：拉取代码有可能产生融合冲突，每次工作前要先拉取最新版本以避免冲突。

4.2 访问项目历史

使用git log查看提交历史。

可以恢复早期版本，方法有

（1）替换为原来某一个版本；

先查看原来版本的hash值和文件名，再用checkout恢复版本。如果要修改最近的提交，可以将hash值用–代替。

（2）撤销修改。

可以反转指定的提交。通过revert可以避免破坏修改历史，但reset会破坏修改历史。

4.3 忽略项目中的文件

在仓库中新建一个名为.gitignore的隐藏文件，文件中输入需要隐藏的文件或文件夹的名称。这样可以避免提交这些文件。
